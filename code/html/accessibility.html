Accessibility

Introduction & color scheme

It is essential to create a website that everyone can use despite disabilities. There are various ways to test if the site meets some WGAC 2.1 requirements. We’ve concluded to use an automated test and manual tests. In this section of our webpage, we will go through how we tested the page, made changes and ensured it was more accessible.

Color scheme

For the color scheme, we researched what colors most people can see. After some research we found that yellow/orange together with blue is a good color scheme because it is visible, and few people have blue – yellow colorblindness. Automated test

For our automated test we used Webaim’s WAVE accessibility checker. This gives us a short summary of each page, as well a detailed section for each element like contrast, alerts, or warnings. When conducting this test, it’s important to keep in mind that the results of the automated test may not be the optimal solution.

Wrong headings

We got zero errors on all of our pages. One of them included using the wrong heading for the title as seen on the picture to the left. The other alert stacks onto the other one, so fixing the first would fix the other one. Instead of using the <p class=”title”> we should’ve used <h1 class=”title”> instead.

Back-to-the-top button

One alert that showed up throughout the whole site was “back to the top” button. This is due to it not being clickable with a keyboard. We have later fixed this by making the text inside the div-button clickable.

Color contrast errors

The only contrast error we faced, was on each of the collapsible buttons. On hover, the background color was turned to bright orange, and the foreground color to dark blue. However, the unicode character is white. We’ve fixed this later on by changing the color of the button content element to the foreground color.

Manual testing & conclusion

In addition to testing accessibility on Webiams website, we also did a multiple of tests manually on our own. Some of the aspects we wanted to test manually was the ability for blind, motor functional disability, and epileptic users to use our website.

Manual test

To check if the website was maneuverable by people with reduced motor functions was use the “tab” and “shift + tab” buttons on the keyboard. This worked well, you can move between the different pages and modify the collapsibles on the page.

The test we used to check if the website had options for blind users was by downloading a text-to-speech extension to Google chrome. This worked verry well on the main content of the page, except the h1 on the “Welcome” page, as well as the nav bar. We could not figure out a reason for this, and the only solution we could find was to add a separate button to the page that would have required a lot of JavaScript.

Since we have a video as a background on the front page, we also added a play/pause button to comfort epileptic users.
